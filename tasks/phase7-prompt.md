## Fixture Generation Strategy

The 3 fixture files must come from **ObjC ANE kernels**, not from Swift — otherwise the "cross-validation" is circular (Swift validating against itself). The correct approach in this repo is:

1. Use the existing ObjC fixture generator (`training/capture_phase6b_goldens.m`) via script:
   ```bash
   ./scripts/capture_phase6b_goldens.sh
   ```
   This writes the 3 `.bin` files into `Tests/ANERuntimeTests/Fixtures/`.

2. **Verify** fixture files exist and are the right size (786,432 bytes each):
   ```bash
   ls -la Tests/ANERuntimeTests/Fixtures/*.bin
   ```

3. **Unlock tests**:
   ```bash
   ANE_HARDWARE_TESTS=1 OBJC_CROSS_VALIDATION=1 swift test \
     --filter test_fwd_attn_numerical_equivalence_with_objc
   ```

### gen_fixtures.m Design Requirements

If you replace the generator in future, the ObjC implementation must:

- Use the **same blob format** as `build_blob()` in `test_full_fused.m` (128-byte header + fp16 weights)
- Use the **same MIL text** as `MILGenerator.fwdAttn(...)`, `MILGenerator.fwdFFN(...)`, `MILGenerator.ffnBwd(...)` for DIM=768, HEADS=12, SEQ=256. You must read the MIL text from the Swift `MILGenerator` target to get it right.
- Set **all weight matrices to 0.01** (constant `w[i] = 0.01f`)
- For `rmsNorm` weights (vectors of length DIM): also set to `0.01f`
- Write IOSurface output using `IOSurfaceLock` / `IOSurfaceUnlock` (matching the SurfaceIO.readFP16 pattern)
- Write fixture as **raw float32 LE** — `float` values, IEEE 754, little-endian (cast float* to uint8_t*, write 4 bytes per element in LSB-first order)
- Accept the output directory as `argv[1]` so the test runner can choose where to write
- Print which files were written and their sizes on stdout for verification

### MIL Text Source

The Swift `MILGenerator` generates the MIL strings. Rather than duplicating them in the fixture generator, the cleanest approach is to:
1. Read `Sources/MILGenerator/` to understand the exact MIL format
2. Use `printf`/`NSString stringWithFormat:` to produce identical MIL in ObjC

Alternatively: run `swift run` or a test to print the MIL text and hard-code it in the generator. Use the approach that's most reliable on this host.

</fixture_generation_strategy>

<cross_validation_methodology>

## Cross-Validation Strategy

### Group 1 — Fixture Binaries (ObjC ANE output, constant weights)
Unblocks the 3 existing skipping tests in `ANERuntimeTests`. These tests compare Swift ANE kernel output against ObjC ANE kernel output — the gold standard is whatever the ObjC kernels produce for identical inputs.

### Group 2 — Golden Stdout (ObjC text output, random/natural weights)
For each ObjC test executable that produces numerical stdout:
1. **Build** via `make -C training` or manual clang
2. **Run** and capture stdout → `training/golden_outputs/<name>.txt`
3. **Write Cross-Validation XCTests** that:
   - Parse numerical values from the golden text files
   - Run equivalent Swift operations
   - Compare within tolerance

### Tolerance Tiers

| Comparison | Tolerance | Rationale |
|---|---|---|
| Forward kernel output (constant weights) | 1e-2 | fp16 has 10 mantissa bits; ANE adds ~1e-3 noise |
| Forward kernel output (random weights) | 1e-2 | Same |
| Backward kernel output | 1e-2 | Backward is more sensitive but still fp16 bounded |
| Weight blob bytes | Exact | WeightBlob.build() must be byte-identical to ObjC build_blob() |
| Causal mask bytes | Exact | CausalMask.blob() must match ObjC encoding |
| Single-step loss | 0.01 | Established in Phase 6 integration test |
| Multi-step loss (5+ steps) | 0.05 | RMSNorm backward divergence accumulates |

### Test Gating Hierarchy
```
swift test                              → 150 tests (pure logic, no hardware)
ANE_HARDWARE_TESTS=1                    → + kernel compile/eval tests
  + OBJC_CROSS_VALIDATION=1            → + fixture-based equivalence tests
ESPRESSO_INTEGRATION_TESTS=1           → + multi-step training tests (needs data)
ESPRESSO_PERF_TESTS=1                  → + benchmark tests (M4+ only)
```

### Known Divergence: RMSNorm Backward

Swift `RMSNorm.backward` is **mathematically correct**; ObjC implementation has an extra `w[i]` factor in the subtraction term. This is intentional and documented in `tasks/lessons.md`. For any comparison involving RMSNorm backward gradients, use tolerance 0.05.

## Capturing Golden Outputs for CrossValidationTests

### Script: `scripts/cross_validate.sh`

Create this script to build + run all ObjC test executables and capture their output:

```bash
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
PROJECT_ROOT="$(dirname "$SCRIPT_DIR")"
TRAINING="$PROJECT_ROOT/training"
GOLDEN="$TRAINING/golden_outputs"

mkdir -p "$GOLDEN"

echo "=== Building probes ==="
make -C "$TRAINING" probes

echo "=== Building manual tests ==="
for name in test_full_fused test_fused_bwd test_fused_qkv test_ane_sdpa5 test_ane_causal_attn test_conv_attn3; do
    xcrun clang -O2 -Wall -Wno-deprecated-declarations -fobjc-arc \
        -o "$TRAINING/$name" "$TRAINING/$name.m" \
        -framework Foundation -framework CoreML -framework IOSurface -ldl
    echo "  Built: $name"
done

echo "=== Capturing golden outputs ==="
for name in test_weight_reload test_perf_stats test_qos_sweep test_ane_advanced \
            test_full_fused test_fused_bwd test_fused_qkv test_ane_sdpa5 \
            test_ane_causal_attn test_conv_attn3; do
    echo "  Running: $name"
    "$TRAINING/$name" > "$GOLDEN/$name.txt" 2>&1 || echo "  WARNING: $name exited non-zero"
    echo "  Captured: $GOLDEN/$name.txt ($(wc -l < "$GOLDEN/$name.txt") lines)"
done

echo "=== Done. Golden outputs in: $GOLDEN ==="
ls -la "$GOLDEN/"
```

### Binary golden outputs for CV-1 and CV-2

For `test_weight_blob_matches_objc` and `test_causal_mask_matches_objc`, you need binary fixtures, not text. Options:
1. Write a small ObjC helper `training/gen_golden_blobs.m` that writes the 4×4 weight blob bytes and seq8 mask blob bytes to `training/golden_outputs/*.bin`
2. OR: read the ObjC source directly and verify the blob format in a pure Swift logic test (no ObjC comparison needed if the format is clear from the source)

Recommended: implement as pure logic tests if the ObjC blob format is already fully documented in the codebase (it is — see `build_blob()` in `test_full_fused.m`). The byte layout is:
- Offset 0: `buf[0]=1`
- Offset 4: `buf[4]=2`
- Offset 64–67: `0xEF 0xBE 0xAD 0xDE` (magic)
- Offset 68: `buf[68]=1`
- Offset 72: `uint32_t` = wsize (= oc*ic*2 bytes)
- Offset 80: `uint32_t` = 128 (data offset)
- Offset 128+: fp16 weights, row-major

This is already tested by `WeightBlob.build()` in ANETypes. CV-1 can be a pure assertion test verifying the blob bytes match the documented format, not a golden-file comparison.

<pitfalls>

## Ranked Pitfalls (Most Likely First)

1. **gen_fixtures.m produces wrong MIL** — if the MIL text in the fixture generator doesn't exactly match what `MILGenerator` produces (byte-for-byte equivalent), the ObjC kernel will compute different values. Fix: read the MIL generator output carefully; use a test to print it and compare.

2. **Fixture bytes written in wrong endianness** — Swift `loadFloat32LEFixture` reads IEEE 754 float bits in LSB-first order. ObjC must write `float` values as 4 bytes LSB-first. On Apple Silicon (little-endian), a simple `fwrite(&f, sizeof(float), 1, fp)` is correct — no byte swapping needed.

3. **IOSurface read in gen_fixtures.m must use Lock/Unlock** — unlike the ANE Swift wrapper which handles locking, direct IOSurface reads in ObjC must call `IOSurfaceLock`/`IOSurfaceUnlock`. Missing this produces garbage values.

4. **SurfaceIO.readFP16 reads from correct position** — the fixture for `fwd_attn` uses `channelOffset: 0` (oOut region). Make sure the ObjC gen_fixtures reads the same region (bytes 0..(dim*testSeqLen*2) in the output surface).

5. **CrossValidationTests target missing dependencies** — needs `ANERuntime`, `CPUOps`, `ANETypes`, `Espresso`, `MILGenerator`. If any are missing, tests will fail to compile or link.

6. **test_full_fused.m and test_fused_bwd.m use srand48(42) weights** — these are NOT the 0.01 constant weights used in the fixture tests. Group 2 CV tests must match the srand48(42) random weight sequence exactly. Reproduce with `srand48(42); drand48()` draw ordering matching the ObjC source.

7. **ANEInterop baseline instability** — `ANE_HARDWARE_TESTS=1 swift test` (full suite without filter) fails on this host at `ANEInteropTests.test_compile_identity_kernel`. Run with `--filter "ANERuntimeTests\|EspressoTests\|CrossValidationTests"` to avoid this known flaky test.
